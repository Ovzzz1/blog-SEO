01

Définition & contexte historique

C'est quoi le DOM ?

Le DOM (Document Object Model) est la représentation en mémoire d'une page web telle qu'un navigateur la construit après avoir chargé et interprété le HTML, le CSS et le JavaScript. C'est un arbre structuré : chaque balise HTML devient un nœud, chaque nœud a des parents, des enfants, des attributs.

⚠ Point crucial

Le DOM n'est pas le HTML source. C'est ce que le navigateur reconstruit à partir du HTML source, potentiellement modifié par le JavaScript.

HTML source (fichier brut)
Ce que le serveur envoie — rien d'autre
Parsing HTML
Construction de l'arbre DOM initial
Exécution JavaScript
Le JS modifie, injecte, supprime des nœuds du DOM
DOM final
Ce que le navigateur affiche — et ce que Google doit rendre

Pourquoi c'est important en SEO ?

Historiquement, Google crawlait uniquement le HTML source. Avec l'explosion des SPAs au milieu des années 2010, une partie croissante du contenu web n'existait plus dans le HTML source — il était injecté dynamiquement par le JS. Google ne voyait plus rien.

Google a développé le Web Rendering Service (WRS) pour rendre les pages. Mais ce rendu a un coût, des délais et des limites — ce qui génère encore aujourd'hui l'essentiel des problèmes d'indexation sur les sites JS-heavy.

02

Fonctionnement technique du WRS

Le pipeline en trois phases

Phase 1 — Crawl HTML (1ère vague)
Googlebot télécharge le HTML brut. Rapide, peu coûteux. Alimente l'index immédiatement.
Immédiat
Phase 2 — File d'attente WRS
Si la page nécessite un rendu JS, elle entre en file. Délai : quelques secondes à plusieurs semaines.
Variable
Phase 3 — Rendu JS & indexation complète (2ème vague)
Le WRS exécute le JS via Chromium evergreen. Le contenu JS-only devient indexable.
Complet

Contraintes du WRS — ce qu'il ne fait pas

Stateless
Cookies, localStorage, sessionStorage : tout effacé à chaque requête. Googlebot ne maintient aucune session.
Pas d'interactions
Aucun scroll, clic ou hover déclenchés. Contenu nécessitant une interaction = jamais indexé.
Timeout ~5 secondes
Si le JS dépasse 5s, le WRS prend un snapshot du DOM partiel. La page peut être indexée incomplète.
Cache JS/CSS 30 jours
Indépendant des directives HTTP. Sans content fingerprinting, tes corrections JS peuvent être ignorées un mois.

Budget de rendu — quelles pages sont réellement rendues ?

Google ne rend pas toutes les pages qu'il crawle. Les signaux qui déterminent si une page passe dans la file WRS :

  • Code HTTP — seules les pages 200 sont garanties d'être rendues.
  • Noindex dans le HTML source — si détecté au crawl, Google peut ne jamais rendre. "noindex = no render".
  • Popularité & autorité — pages très linkées ou fréquemment mises à jour, traitées en priorité.
  • Performance serveur — un serveur lent réduit le crawl rate et le budget de rendu.
03

La limite des 2 Mo — nouveauté mars 2026

2 Mo

La troncature silencieuse de Googlebot

Confirmée par Google Search Central en mars 2026. Googlebot ne récupère que les 2 premiers Mo de tout fichier HTML ou ressource textuelle. Tout ce qui est au-delà est purement ignoré — sans aucune notification, sans erreur visible.

Ce qui se passe concrètement

Si un fichier dépasse 2 Mo après décompression, Googlebot arrête le téléchargement exactement à 2 Mo et traite ce contenu tronqué comme le fichier complet. Cette limite s'applique individuellement à chaque ressource. Un bundle JS de 2,5 Mo sera tronqué, produisant une erreur de syntaxe JS — la page peut apparaître blanche pour Googlebot.

Risque Ce qui se passe Impact SEO
JSON-LD en bas de <body> Poussé après les 2 Mo sur les pages lourdes Perte des rich snippets
Liens de footer Maillage interne non lu Pages orphelines
CSS inline excessif Pousse le contenu textuel hors zone indexée Contenu principal ignoré
Images Base64 inline Grossissent massivement le HTML Même effet que CSS inline
Gros bundles JS Troncature = erreur de syntaxe Page blanche pour Googlebot
✓ Règle pratique

Maintenir le HTML source sous 1,5 Mo pour avoir une marge de sécurité. Placer les éléments critiques (canonical, noindex, title, structured data) le plus haut possible dans le <head>.

04

DOM vs HTML source — ce que Google lit vraiment

Meta robots / noindex Immédiate Honorée après rendu Le noindex source prime — peut bloquer le rendu
Canonical Très prioritaire Risquée Ne jamais modifier via JS si déjà dans le source
Hreflang Recommandée Risquée Indispensable dans le source — validation bidirectionnelle
Structured data Immédiate Supportée (JSON-LD) Valide si non tronquée par la limite 2 Mo
Liens internes Découverte rapide Après rendu JS-only = retarde le crawl profond

Le canonical JS — pourquoi c'est risqué

Google lui-même qualifie la gestion des canonicals via JS de "risquée" dans sa documentation. En cas de conflit entre un canonical HTML source et un canonical modifié via JS, Googlebot tend à favoriser la version HTML source — ou peut être "désorienté", entraînant une indexation imprévisible.

❌ Cas concret à éviter

Une SPA dont le canonical du HTML source pointe toujours vers la home (jamais mis à jour lors des soft navigations). Google consolide tout le contenu du site sur la home → cannibalisation généralisée.

Le noindex JS — une fenêtre d'indexation involontaire

Scénario fréquent et dangereux : une page a un <meta name="robots" content="noindex"> injecté uniquement via JavaScript.

  1. Crawl HTML source → pas de noindex → Google peut indexer immédiatement
  2. File WRS (délai variable) → DOM avec noindex → Google le voit enfin
  3. Pendant cette fenêtre, la page peut apparaître dans les SERPs

Le hreflang JS — problème de validation bidirectionnelle

Les hreflang reposent sur une confirmation bidirectionnelle. Si les signaux sont uniquement dans le DOM, Google doit effectuer deux passages de rendu sur deux URLs différentes pour valider la relation. Résultat : affichage de la version localisée correcte retardé de plusieurs semaines dans les SERPs.

CSR — l'impact concret

Le serveur envoie un HTML quasi-vide. Si le bundle JS échoue, si l'API est trop lente ou si Googlebot subit un timeout, la page peut être indexée sans aucun contenu textuel. De nombreux sites React/Vue ont constaté des chutes de trafic après des migrations sans SSR.

HTML source d'un site CSR
<body>
  <div id="root"></div>
  <script src="/bundle.js"></script>
</body>
<!-- Googlebot voit : rien -->

ISR — le point de vigilance

L'architecture de référence en 2026 pour le e-commerce. Mais les TTL de régénération doivent être maîtrisés : si Googlebot crawle pendant la fenêtre de régénération, il peut recevoir une version périmée.

06

Hydration mismatches — le bug SEO invisible

L'hydratation est le processus par lequel le JS "réveille" le HTML statique serveur pour le rendre interactif. Un mismatch survient quand le DOM côté client diffère de celui envoyé par le serveur.

Causes fréquentes documentées

A Données temporelles

Afficher "il y a 5 minutes" côté serveur et côté client avec des horloges décalées → mismatch immédiat. Le serveur génère une valeur, le client recalcule au montage et obtient une autre.

B APIs navigateur dans le rendu initial

Utiliser window.innerWidth, localStorage ou navigator dans la logique qui génère le HTML initial. Ces APIs n'existent pas côté serveur → crash ou valeur différente.

C Structure HTML invalide

Un <div> imbriqué dans un <p> par exemple. Le navigateur corrige automatiquement la structure, créant un DOM différent du HTML source.

Impact SEO concret

Sur un mismatch sévère, React ou Vue peuvent décider de détruire le DOM existant et le reconstruire entièrement côté client. Les Core Web Vitals dégringolent (surtout l'INP), et si des éléments SEO critiques sont perdus pendant la reconstruction, Googlebot peut indexer une version incomplète.

💡 Diagnostic

Les frameworks loggent les mismatches dans la console navigateur. En audit : activer la capture des erreurs console dans Screaming Frog ou un headless browser.

08

Lazy loading & content-visibility

IntersectionObserver — ce que Googlebot fait vraiment

Googlebot tente de simuler un viewport très long pour déclencher les événements IntersectionObserver sans défiler. Ça fonctionne dans la plupart des cas pour les images et blocs standards.

Mais : Googlebot ne déclenche aucun événement de scroll, clic ou survol. Contenu nécessitant une interaction explicite = jamais indexé.

✅ Sûr
loading="lazy" natif sur les images. Parfaitement compris par Googlebot. Méthode recommandée.
❌ Risqué
Contenu textuel critique ou liens importants derrière un lazy-load JS custom. À ne jamais faire.

CSS content-visibility: auto — lazy rendering ≠ lazy loading

Point souvent mal compris : content-visibility: auto ignore le rendu visuel des éléments hors-écran mais le contenu reste présent dans le DOM et est parfaitement indexable par Googlebot. C'est du lazy-rendering, pas du lazy-loading.

✓ Confirmé par John Mueller

content-visibility n'influence pas directement le classement mais améliore les scores de Core Web Vitals (INP, LCP). Aucun risque SEO à l'utiliser.

09

Signaux conflictuels & comportement de Google

Ressources JS/CSS bloquées dans robots.txt

Bloquer les fichiers JS/CSS dans le robots.txt empêche Googlebot de les télécharger → rendu partiel ou nul. C'était une "optimisation de crawl budget" répandue il y a quelques années. Aujourd'hui c'est une erreur grave.

💡 Diagnostic

Search Console → Inspection d'URL → "Plus d'informations" → ressources bloquées. Screaming Frog identifie aussi les ressources bloquées lors d'un crawl JS.

Quand Google ignore le rendu

  • Budget de rendu insuffisant (pages peu valorisées sur des gros sites)
  • Timeout de 5 secondes dépassé
  • Erreurs JS bloquantes (syntaxe, modules manquants)
  • Ressources bloquées dans robots.txt
  • Troncature à 2 Mo d'un bundle JS critique

Screaming Frog : crawl standard + crawl JS rendering. Comparer titles, H1, nb liens, canonical. Divergence > 20% = signal d'alerte.

L'outil de référence. Voir le HTML rendu tel que Google l'a réellement vu. Différent de Chrome DevTools (DOM live après toutes les interactions).

Si HTML source > 1,5 Mo : risque de troncature à 2 Mo. Systématiser cette vérification dans tout audit technique depuis 2026.

Erreurs fréquentes — par profil

Erreurs de développeurs

  • Meta via useEffect — balises head injectées après le montage. HTML source vide au crawl.
  • Blocage JS/CSS dans robots.txt — héritage d'une époque révolue. Casse le rendu de Googlebot.
  • Pas de content fingerprinting — bundles JS nommés app.js sans hash. Cache WRS 30 jours. Corrections JS invisibles pendant un mois.
  • HTML > 1,5 Mo — CSS inline massif, images Base64, JSON inline trop lourds. Contenu indexable poussé après le seuil de troncature.

Erreurs de SEOs débutants

  • "Google exécute le JS donc ça va" — le délai WRS, le budget de rendu, les erreurs JS, la limite 2 Mo limitent la portée réelle du rendu.
  • Chrome DevTools comme référence — Inspect montre le DOM live. La vraie référence : HTML rendu dans GSC.
  • Prerendering comme solution durable — officiellement obsolète selon Google. Risque de cloaking si les versions divergent.

Erreurs de SEOs intermédiaires

  • "Le SSR résout tout" — un hydration mismatch sévère peut reconstruire le DOM côté client et écraser les balises head serveur.
  • Ignorer le délai WRS dans les stratégies de contenu — sur un site CSR, l'indexation peut prendre plusieurs semaines.
  • Ne pas vérifier la taille des fichiers en audit — depuis mars 2026, vérification systématique obligatoire.
  • Hreflang en JS sur des sites internationaux — délai de double rendu pour la validation bidirectionnelle = plusieurs semaines de décalage.

Soft navigations & SPAs — statut 2026

Comment Googlebot gère les SPAs

Googlebot explore chaque URL indépendamment, en simulant un rechargement complet à chaque fois. Il ne "suit" pas les soft navigations JS. Si le HTML source de chaque URL est identique (balises head non mises à jour côté serveur), Google reçoit le même contenu pour toutes les URLs de la SPA.

L'API Soft Navigations — statut expérimental

Google mène depuis 2025 un Origin Trial pour mesurer les Core Web Vitals sur chaque transition virtuelle d'une SPA, pas seulement sur le chargement initial.

MétriqueDescriptionStatut 2026
navigationId Identifiant unique pour chaque transition virtuelle Expérimental
interaction-contentful-paint Nouveau LCP mesuré après une interaction utilisateur Expérimental
Attribution par route Segmentation des CWV par URL virtuelle (/produit/123) En test
💡 Signal à surveiller

Ces données ne sont pas encore des facteurs de classement directs dans le rapport CrUX officiel. Mais la direction est claire : Google veut évaluer la qualité de l'expérience sur l'ensemble d'une session SPA. À intégrer dans le radar pour 2027.

13

Edge cases & situations avancées

🔬 Edge cases

Cas réels mais rares. Là pour anticiper, pas pour être la préoccupation principale sur 90% des projets.

Shadow DOM : open vs closed

Googlebot effectue un "aplatissement" (flattening) du DOM lors du rendu des Web Components : il fusionne le Shadow Tree avec le Light DOM principal. En mode open, le shadow DOM est accessible → Googlebot le lit correctement. En mode closed, le moteur Chromium peut techniquement l'aplatir, mais c'est moins fiable. Mode open recommandé pour tout contenu à indexer.

Si on utilise des <slot> dans des Web Components : s'assurer que le contenu du Light DOM est correctement positionné. Contenu non slotté = potentiellement ignoré.

Service Workers et Googlebot

Googlebot n'installe pas durablement les Service Workers — il les contourne pour récupérer la version fraîche directement sur le serveur. Un Service Worker configuré pour enrichir ou transformer le contenu (A/B test, personnalisation) sera simplement bypassé par Googlebot.

⚠ Règle

Ne jamais compter sur un Service Worker pour délivrer du contenu essentiel à Googlebot. Toujours utiliser l'amélioration progressive.

ISR et double cache pendant la régénération

En ISR (Next.js), deux versions d'une page peuvent coexister simultanément pendant la régénération. Si Googlebot crawle pendant cette fenêtre et que le canonical de la nouvelle version pointe différemment, il reçoit des signaux contradictoires par rapport à son crawl précédent. Sur des sites à fort volume ISR avec des TTL courts (quelques minutes), ce pattern peut créer des incohérences difficiles à diagnostiquer — et qui ne se reproduisent pas à l'inspection manuelle.

Règles de décision rapide

Architecture de rendu
Site à enjeu SEO = SSR ou SSG. CSR sans prerendering à proscrire. Prerendering = transitoire acceptable, officiellement obsolète, risque cloaking.
Taille des fichiers
HTML source < 1,5 Mo. Éléments critiques le plus haut dans le <head>. Content fingerprinting obligatoire sur les bundles JS/CSS.
Balises critiques
Canonical, noindex, hreflang = HTML source. Jamais JS-only. Title et meta description = mécanismes natifs du framework SSR côté serveur.
Diagnostic
HTML source ≠ DOM rendu ≠ ce que Google voit. Tester les trois séparément. Référence : inspection URL GSC, pas Chrome DevTools.
Timing des corrections
Corrections rendu JS → 2 à 6 semaines dans GSC. Cache WRS → 30 jours max. Ne pas sur-corriger pendant la fenêtre d'attente.
Scripts tiers
Jamais conditionner l'injection de contenu SEO à l'init d'un script tiers. async/defer toujours. Ne jamais compter sur un Service Worker pour Googlebot.
